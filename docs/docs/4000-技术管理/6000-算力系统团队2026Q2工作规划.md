## CoreMind Token服务平台

### NewAPI网关会话保持能力

**背景与目标**

在多轮对话场景中，`LLM`推理引擎会将历史`KV Cache`（Key-Value Cache，注意力机制中的键值对缓存）保留在显存中以加速后续推理。若同一会话的请求被路由至不同的后端推理实例，已缓存的`KV Cache`将无法被复用，引擎需要重新计算历史上下文，造成算力浪费，并显著增加首`Token`延迟（`TTFT`）。

通过在`NewAPI`网关层引入**会话保持（Session Affinity）**机制，可将同一会话的连续请求持续路由至同一推理实例，从而最大化`KV Cache`命中率，提升服务吞吐量与响应质量。

**解决问题**

- 提升推理引擎侧`KV Cache`利用率，减少冗余重计算；
- 降低多轮对话场景下的端到端响应延迟；
- 在不增加硬件资源的前提下提升整体服务质量与效率。

**具体事项**

| 事项 | 说明 |
| --- | --- |
| 会话标识设计 | 基于`session_id`或`conversation_id`字段标识会话，支持客户端通过请求头或请求体传入 |
| 路由策略改造 | 改进`NewAPI`网关转发逻辑，依据会话标识将请求哈希映射至固定后端实例 |
| 实例失效处理 | 后端实例不可用时自动降级至其他实例，并清除对应会话绑定关系，保障服务可用性 |
| 配置化支持 | 支持按模型或渠道维度开启/关闭会话保持，满足不同场景的差异化需求 |

### 动态模型服务分组路由策略

**背景与目标**

`CoreMind Token`服务平台目前同时接入了公司自建模型服务（内网`NPU`/`GPU`推理集群）与第三方模型服务（如阿里云`百炼`等外部`API`）。自建服务在成本和数据安全方面具有明显优势，应作为优先调用来源。然而，当自建服务出现高延迟或不可用时，若无自动切换机制，将直接影响业务连续性。

需要建立一套动态分组路由策略，实现**优先自建、智能降级、自动恢复**的流量调度能力。

**解决问题**

- 在自建模型服务可用时，优先路由至自建服务，降低外部`API`调用成本；
- 当自建服务响应延迟超阈值或健康检测失败时，自动切换至第三方服务，保障业务可用性；
- 服务恢复后自动将流量切回自建服务，减少对外部`API`的依赖。

**具体事项**

| 事项 | 说明 |
| --- | --- |
| 路由切换策略设计 | 以`TokenKey`（渠道/`API Key`）为维度，基于平均响应延迟（P99）、错误率、可用性等指标制定切换规则，支持配置阈值 |
| 健康检测机制 | 对各后端服务实例进行主动健康探测（`Active Health Check`），支持`HTTP`探活和模型推理探活两种模式 |
| 动态分组切换 | 实现基于指标的自动分组切换能力，支持平滑切流，避免流量突变导致后端压力骤升 |
| 监控集成 | 调研并集成`Uptime Kuma`实现服务可用性持续监控，提供可视化大盘与告警通知能力 |
| 切换日志审计 | 记录每次路由切换事件，包含切换原因、时间戳及影响范围，便于事后分析与复盘 |

### 核心服务高可用建设

**背景与目标**

`CoreMind Token`服务已成为公司`AI`基础设施的关键组成部分，承载了内部所有业务系统调用大模型`API`的核心链路。服务中断将直接影响公司各部门的`AI`工具使用及相关业务流程。

目前`NewAPI`网关与`OIDC`认证服务已完成高可用改造，Q2需重点完成剩余依赖组件的高可用部署，并建立完善的监控告警体系。

**解决问题**

- 消除服务架构中的单点故障（`SPOF`），提升整体服务可用性（目标`SLA ≥ 99.9%`）；
- 建立全链路监控告警，实现故障的快速发现与响应；
- 完善关键业务指标的可观测性，支撑后续容量规划与成本优化决策。

**具体事项**

**高可用部署**

| 组件 | 当前状态 | 目标方案 |
| --- | --- | --- |
| `NewAPI`网关 | ✅ 已完成 | 多副本部署 + 负载均衡 |
| `OIDC`认证服务 | ✅ 已完成 | 多副本无状态部署 |
| `PostgreSQL`数据库 | 🔲 待完成 | 主从流复制 + 自动故障转移（`Patroni`/`Repmgr`） |
| `Redis`缓存服务 | 🔲 待完成 | 哨兵模式（`Sentinel`）三节点部署，实现主从切换 |

**监控告警建设**

| 监控维度 | 具体指标 | 告警策略 |
| --- | --- | --- |
| 组件健康状态 | 各组件`Pod`存活率、`CPU`/内存使用率、磁盘`IO` | 关键组件存活率低于阈值时立即告警 |
| Token服务核心指标 | 平均`RPM`（每分钟请求数）、`TPM`（每分钟`Token`数） | 指标异常波动超基线30%时触发告警 |
| `API`可用性 | 各模型渠道的成功率、错误率、响应时延（P50/P99） | 错误率超阈值或延迟突增时告警 |
| 上游大模型服务延迟 | 自建模型与第三方模型接口响应延迟 | 延迟超预设阈值时联动动态路由切换 |
| `Uptime Kuma`监控 | 全链路服务可用性端到端探测 | 服务中断时立即推送告警通知 |

### 用量统计与用户组织管理

**背景与目标**

**短期目标**：`CoreMind`现有用户与渠道管理基于"分组"维度，但用户数据中缺少所属部门字段，导致无法将公司组织架构与系统分组关联，进而无法按部门维度统计模型调用量与`Token`消耗情况，影响资源分配决策与成本归因。

**长期目标**：随着使用场景的持续扩展（如企业级`Agent`应用、多部门私有模型接入等），需支持基于组织架构对渠道、模型及用量额度进行精细化管控。具体场景包括：公用内部部署模型、部门专属私有模型、外部渠道模型的差异化访问控制等。

**解决问题**

- 补全用户组织信息字段，建立"组织架构 ↔ 分组"的映射关系，支撑部门维度的用量统计；
- 提供灵活可配置的渠道、模型及额度管控能力，以适应不断扩展的使用场景；
- 沉淀精细化的用量数据，为后续资源规划和价值衡量提供数据基础。

**具体事项**

**【用户管理】模块增强**

| 功能点 | 说明 |
| --- | --- |
| 组织信息字段 | 用户数据新增所属部门/组织字段，支持与公司`LDAP`/`SSO`目录同步 |
| 批量操作支持 | 支持批量导入、批量修改用户分组、批量调整用量额度等运营操作 |
| 数据导出 | 支持按条件筛选导出用户列表，格式支持`CSV`/`Excel`，便于数据审计 |

**【用量统计】模块建设**

| 功能点 | 说明 |
| --- | --- |
| 统计维度 | 支持按模型、渠道、分组、用户（组织）及时间区间进行多维度统计 |
| 基础指标 | 请求次数（`RPM`累计）、`Token`消耗量（输入/输出分别统计）、额度使用率 |
| 数据看板 | 控制台保留现有"数据看板"；管理后台新增独立"用量统计"模块，支持图表与数据表切换 |
| 权限隔离 | 部门管理员仅可查看本部门数据，平台管理员可查看全局数据 |

## 推理调度引擎

### 实现推理服务分布式跨节点部署能力

**背景与目标**

公司计划基于`NPU`（华为昇腾`910B3`，每节点8卡）部署`GLM-5 744B`超大规模语言模型。受限于单节点`NPU`显存容量，单节点8卡无法完整加载`744B`参数量的模型权重，需采用**跨节点张量并行（`TP16`）**策略，通过2个节点共16张`NPU`卡协同承载一个完整模型实例。

此外，`PD`分离（`Prefill-Decode Separation`）作为提升推理服务吞吐量与降低首`Token`延迟的关键技术，需要完成产品化落地，进入可运营状态。

**解决问题**

- 解决超大规模模型（`744B`）因单节点显存不足无法部署的问题；
- 通过跨节点高速互联（`InfiniBand`/`RoCE`）实现张量并行通信，最大化`16`卡算力利用率；
- `PD`分离产品化，将`Prefill`（预填充）与`Decode`（生成解码）阶段分配至专属硬件资源，针对性优化`TTFT`（首`Token`延迟）与`TPOT`（每输出`Token`时延），提升整体服务效率。

**具体事项**

| 事项 | 说明 |
| --- | --- |
| `TP16`跨节点部署 | 实现张量并行度为16的跨节点分布式推理部署，2节点×8卡协同运行`GLM-5 744B`模型 |
| 跨节点通信优化 | 配置并验证节点间高速互联网络（`InfiniBand`/`RoCE`），优化`AllReduce`等集合通信操作的延迟与带宽 |
| `PD`分离产品化 | 将`Prefill`实例与`Decode`实例独立部署，支持差异化资源配置（`Prefill`计算密集型，`Decode`内存带宽密集型），完成调度层适配 |
| 稳定性验证 | 对跨节点推理架构进行压测，验证在高并发场景下的吞吐量、延迟及节点容错能力 |
| 运维支撑 | 建立分布式推理实例的统一监控视图，支持节点级别的健康检测与异常告警 |

## 成本统计

### 完善模型计量统计能力

**背景与目标**

现有成本统计体系主要覆盖文本生成类`LLM`的`Token`消耗统计，但随着平台上`Embedding`（向量化）、`Reranker`（重排序）等非生成类模型的接入规模扩大，这些模型的调用量与资源消耗缺乏精准计量，导致成本归因不完整，影响资源优化与定价决策。

**解决问题**

- 扩展计量维度，覆盖`Embedding`模型（按向量化文本长度计量）、`Reranker`模型（按排序候选数量计量）等非生成类模型；
- 建立统一的模型调用计量标准，为跨模型类型的成本比较与资源决策提供数据支撑；
- 为后续按调用成本进行部门级成本分摊提供数据基础。

**具体事项**

- 梳理`Embedding`、`Reranker`等模型的计量维度，制定统一的计量单位规范；
- 在`CoreMind`平台中实现对上述模型调用量的精准采集与存储；
- 将新增计量数据接入用量统计模块，支持成本分析报表输出。

## 战略方向探索

### Agentic AI基础设施转型探索

**背景与目标**

随着`Agentic AI`（自主智能体`AI`）技术的快速演进，以`LLM`为核心的自主决策、工具调用、多步推理等能力正在重塑`AI`应用范式。团队需要前瞻性地调研行业痛点与新兴方向，结合自身算力基础设施优势，探索向`Agentic AI Infra`方向的转型路径，以应对行业变化并支撑团队长期发展。

**具体事项**

- 调研`Agentic AI`基础设施领域的行业痛点与技术趋势（包括`Multi-Agent`框架、工具编排、长上下文内存管理、`Agent`可观测性等方向）；
- 结合团队现有技术栈，评估可切入的细分方向与可行性；
- 至少完成一个新方向的深度调研与原型验证，并在团队内进行技术分享。

### 利用AI技术提升团队研发效能

**背景与目标**

团队人力资源结构的变化要求个人产出效率持续提升。`AI`辅助编程、智能代码审查、自动化文档生成等工具正在深刻改变软件研发模式。团队需系统性地调研并落地`AI`研发工程化最佳实践，以`AI`驱动个人与团队效能的实质性提升。

**具体事项**

- 调研行业主流`AI`开发工具（如`GitHub Copilot`、`Cursor`、`Windsurf`等）及工程化最佳实践，评估团队适用性；
- 在至少一个项目（如`CoreMind`）中落地完整的`AI`工程化实践，涵盖需求理解、代码生成、测试用例生成、代码审查等环节；
- 沉淀团队`AI`研发工程化实践规范，形成可复用的方法论与工具链推荐。
